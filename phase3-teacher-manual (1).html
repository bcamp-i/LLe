Phase goal

By the end of these three weeks, each student's machine should feel distinctly theirs. The desktop looks the way they chose. The terminal behaves the way they configured. The shell has their aliases, their prompt, their functions. And the machine runs their first real scripts automatically.

Phase 3 is the course's inflection point. The first two phases were about learning to operate a system someone else built. Phase 3 is about making it yours. The difference between a user and someone who truly owns a machine is whether they understand and can modify the layer between the OS and their daily workflow. That layer is what this phase teaches.

Three things to watch for across this phase. First: students who over-customize without understanding what they changed — pull them back to reading the config files, not just clicking settings. Second: students who are intimidated by scripting — the warm-up for session 49 directly addresses this by connecting scripts to work they have already done. Third: students who finish early — give them the next script challenge, not free time. The scripting sessions in week 13 have an almost infinite ceiling for depth.

Session 41
KDE Plasma — understanding the desktop as software
10 minwarm-up
20 minconcept
25 minexercise
5 minclose
Overview

The goal of this session is a single conceptual shift: the desktop is not the operating system. It is a software layer, as replaceable and configurable as any other. KDE Plasma is one possible desktop shell among several. Understanding this means understanding that "how the desktop looks" and "how the OS works" are completely separate concerns.

This is also the session where students first interact deeply with KDE's configurability. The aim is not to configure everything — it is to understand that everything can be configured, and to know where to find the controls.

Warm-up · 10 min

Ask the class: "What parts of this desktop do you interact with every day?" Build a list on the board together — taskbar, clock, application launcher, system tray, notifications, window title bars, wallpaper, file manager, desktop right-click menu. When the list is complete, ask: "Which of these do you think you can change?" The answer to every item is yes. Then ask: "Which of these do you think you could remove entirely?" Also yes, for most of them. This reframes the desktop from a fixed environment to a configurable one before the concept section begins.

Core concept · 20 min

KDE Plasma is a desktop shell — software that provides the graphical interface between the OS and the user. It is separate from the Linux kernel, separate from the display server, and separate from the applications that run inside it. It can be removed and replaced. On the same Kubuntu installation, you could install GNOME Shell or XFCE and switch between them at login.

KDE Plasma is composed of distinct components. Understanding what each one does makes the System Settings application comprehensible instead of overwhelming:

KDE Plasma components — draw on board
KWin — window manager: opens, closes, resizes, tiles windows Plasma Shell — the panel, widgets, desktop, app launcher KDE Compositor— transparency, blur, shadows, animations Dolphin — file manager (separate application) System Settings— configuration front-end for all of the above All of these talk to each other but are separate programs. Each has its own config files in ~/.config

KWin is the window manager. It decides where windows appear, how they are resized, whether they can be tiled, and what happens when you press keyboard shortcuts related to windows. KWin has its own scripting API — you can write JavaScript to automate window behavior. We will not go that deep, but knowing it exists changes how students think about what is possible.

The compositor handles visual effects — the blur behind the panel, the transparency of windows, the animations when windows open and close. It sits between the applications and the display. On machines with weak GPUs, disabling the compositor entirely makes everything faster. On machines with good GPUs, the compositor adds useful visual feedback without cost.

Widgets are small applications that live on the panel or the desktop. The clock is a widget. The system tray is a widget. The application launcher is a widget. You can add, remove, rearrange, and resize widgets. KDE's widget store (Get New Widgets) provides hundreds of community-made ones.

Exercise · 25 min

Guided exploration — understand before you change:

  1. Right-click the desktop. What options appear? Select "Add Widget" — browse the available widgets without adding anything yet. What categories exist?
  2. Right-click the taskbar panel. Select "Show Panel Configuration." Unlock it. Drag the panel to the top of the screen. Resize it. Move it back. Re-lock it. What changed in the configuration dialog between locked and unlocked states?
  3. Open System Settings. Spend 5 minutes browsing every category — do not change anything, just read the category names and the first level of options inside each. Write down 3 settings you want to change and what they control.
  4. Open System Settings → Window Management → KWin Scripts. What scripts are listed? What do they claim to do?
  5. Open System Settings → Shortcuts → Global Shortcuts. Find the shortcut to open the terminal (Konsole). What is it currently? Change it to a shortcut you will remember. Test it.
  6. Find the option to enable window tiling. Enable it and test: drag a window to the left edge of the screen. What happens?
Expected student questions
Q: Can I install a completely different desktop on this machine?
A: Yes. sudo apt install ubuntu-gnome-desktop installs GNOME. You choose which desktop at the login screen. Multiple desktops can coexist — they share the same files and packages but provide different interfaces. Some packages install differently on each desktop (GNOME apps on KDE, KDE apps on GNOME) but they all run.
Q: Is KDE better than GNOME?
A: Different design philosophy, not a ranking. KDE prioritizes configurability — almost everything can be changed. GNOME prioritizes simplicity and consistency — fewer options, more opinionated defaults. XFCE prioritizes performance — lightweight, fast, no compositing by default. Each is the right choice for different users and contexts. The fact that you chose KDE does not mean GNOME is wrong for someone else.
Q: What happens if I break the desktop?
A: The terminal still works — Ctrl+Alt+F2 opens a text console with no desktop at all. From there you can fix configuration files, reinstall the desktop, or undo changes. The OS and your files are completely independent of the desktop. This is an important safety net that does not exist on Windows or macOS.
Close · 5 min

Name the difference between the panel, a widget, and the window manager. What is KWin responsible for? What does the compositor add? Preview session 42: "Now that you know what each part is, tomorrow we configure it deliberately."

The Ctrl+Alt+F2 escape hatch mentioned in the Q&A is worth demonstrating briefly if time allows. Showing students that the terminal still exists behind the desktop — and that they can recover from a broken desktop without losing files — removes a significant source of anxiety about customization. Students who know there is an escape hatch experiment more boldly.

The widget store exercise (task 1) should be browsing only — do not let students spend the session installing widgets. The goal is understanding the architecture, not decoration. Save widget installation for students who finish early.

Session 42
Personalizing KDE — themes, fonts, shortcuts
5 minwarm-up
15 minconcept
35 minexercise
5 minclose
Overview

Deliberate customization with understanding, not random clicking. Every change a student makes today should be intentional, and they should be able to answer: what did I change, where is the setting, and where is the configuration file that stores it? The second question — where is the file — connects this session directly to week 12's dotfiles content.

Warm-up · 5 min

Ask: what would make your desktop faster to use, not just prettier? Focus the class on functional customization — keyboard shortcuts that replace mouse clicks, panel layout that matches workflow, virtual desktops that separate contexts — rather than aesthetic choices. Both matter, but functional changes compound. A shortcut used 20 times per day saves minutes per day, hours per month.

Core concept · 15 min

KDE System Settings is a front-end for configuration files. Every setting you change through the UI writes to a plain text file somewhere in ~/.config or ~/.local/share. This means: your entire KDE configuration is a collection of text files in your home directory. You can back it up with a file copy. You can restore it by copying the files back. You can share it with another Kubuntu machine and get an identical environment.

Navigate the settings by function, not by exploration:

Appearance — global theme (affects everything at once), application style (Qt widget appearance), plasma style (panel and widgets), colors, fonts, icon theme, cursor. Apply a global theme if you want a complete change, or mix components from different themes for a custom look.

Shortcuts — global shortcuts apply system-wide regardless of which application is focused. Custom shortcuts can run any command. This is how you bind a key to open your most-used applications without clicking the launcher.

Virtual Desktops — multiple workspaces on the same monitor. Assign a shortcut to each. Keep code on desktop 1, browser on desktop 2, terminal on desktop 3. Switch instantly with the keyboard. Virtual desktops are a productivity multiplier once the habit forms — they take about 3 days to feel natural.

Exercise · 35 min

Configure with intent. Before changing anything, write what you are about to change and why.

  1. Choose and apply a global theme. After applying: ls ~/.local/share/plasma/desktoptheme/ — what appeared there?
  2. Set up at least 2 custom keyboard shortcuts for applications you use often. Test each one.
  3. Configure 3 virtual desktops with meaningful names (not just 1, 2, 3). Set keyboard shortcuts to switch between them. For the rest of this session, use virtual desktops — browser on one, terminal on another, files on a third.
  4. Change the font used for interface elements. Open a file manager and a text editor to see the effect. Set it back if you dislike it — the point is understanding where the setting is.
  5. Open a terminal and run: ls -lt ~/.config/ | head -10. This shows the 10 most recently modified files in your KDE configuration directory. Which files changed after your System Settings adjustments? Open one with cat — what format is it?
  6. Write a one-paragraph description of the customizations you made and why you chose them. Save it to ~/projects/linux-course/phase3/kde-config.md.

global themeapplication style plasma styleicon theme virtual desktopcustom shortcut ~/.config~/.local/share font rendering
Teacher note

Task 5 — finding the modified config files after making settings changes — is the bridge between this week and next week. When a student sees that changing the color scheme wrote to ~/.config/kdeglobals and then opens that file and reads it, the connection between UI and text configuration becomes concrete. Point this out explicitly: "You just made this change through a settings dialog. You can also make it by editing that file directly. Both are the same thing." This prepares the week 12 concept without lecturing about it yet.

Virtual desktops reliably divide the class: students who take to them immediately and will use them forever, and students who find them confusing and will disable them. Both are valid responses. The point is exposure — some will adopt this workflow in week 11 and compound the benefit over 3 months of the course.

Session 43
Konsole and terminal customization
5 minwarm-up
15 minconcept
35 minexercise
5 minclose
Overview

Students will spend a significant portion of the next three months in the terminal. A well-configured terminal is measurably faster and more comfortable to use. This session is the one-time investment that pays dividends in every session that follows.

Warm-up · 5 min

Ask: how many hours per week do you think you will spend in the terminal for the rest of this course? Rough estimates only. Then: if a configuration change saves you 5 seconds per interaction, and you have 100 interactions per session, that is 8 minutes per session. Over the remaining 56 sessions: roughly 7 hours. A 30-minute investment now is worth doing. That calculation is the warm-up.

Core concept · 15 min

Konsole profiles allow multiple saved terminal configurations. Each profile specifies: the shell to launch, the starting directory, the font, the color scheme, the cursor style, the scrollback history length, and several other options. You can create a "course" profile for this class, a "reading" profile with larger text, and switch between them per-tab. Profiles are stored in ~/.local/share/konsole/ as .profile files — plain text, copyable.

Color schemes affect readability in practical ways. Stderr (errors) is traditionally shown in red. A color scheme with low contrast between red and white text makes errors harder to spot. Good schemes for terminal use: Solarized Dark (warm tones, very readable for long sessions), Dracula (high contrast, popular in editors), One Dark (neutral, works well with syntax-highlighted output). Choose based on how easy it is to visually distinguish: the command you typed, normal output, errors, and the prompt.

The PS1 variable controls the shell prompt — the text that appears before every command. The default on Kubuntu is something like username@hostname:~$. PS1 is a string that bash interprets each time it displays the prompt. Special escape sequences: \u username, \h hostname, \w full current path, \W current directory name only, \$ $ for regular user, # for root. Color codes use ANSI escape sequences: \[\033[CODEm\]text\[\033[0m\] where CODE is a number (31=red, 32=green, 33=yellow, 34=blue, 35=purple, 36=cyan, 1=bold). The \[ and \] around color codes tell bash not to count the invisible characters when calculating the cursor position — missing them causes wrapping bugs.

Exercise · 35 min
  1. Open Konsole settings (Settings → Edit Current Profile). Create a new profile called "course". Set your preferred monospace font at minimum 12pt — the text you will read for 3 months should be comfortable. Apply it.
  2. Try at least 3 color schemes from the Color Scheme section. Choose one and write why: ease of reading, contrast, aesthetics. Apply it to your course profile.
  3. Open ~/.bashrc with nano. Find the line defining PS1 (search with Ctrl+W). If it is commented out or absent, add one. Build a PS1 that includes at minimum the current directory and a clear separator. A starter: PS1='\[\033[32m\]\u\[\033[0m\]:\[\033[34m\]\w\[\033[0m\]\$ ' — green username, blue path. Save, then run source ~/.bashrc. Does the prompt update immediately?
  4. Add at least one useful alias to ~/.bashrc: alias ll='ls -la' at minimum. Source the file. Test the alias. Open a new terminal — does the alias work there too?
  5. Change Konsole's title bar to show the current directory (Settings → Tab Bar/Splitter → Tab title format). Navigate to a few directories and watch the title bar update.
  6. Take a screenshot with Spectacle (Search for "Spectacle" in the app launcher). Save it to ~/projects/linux-course/phase3/terminal-setup.png. This is your before/after reference for the end of the course.
Vocabulary
Konsole profilecolor scheme PS1\u \h \w \W ANSI color codes\[ \] (non-printing) source ~/.bashrcalias ~/.local/share/konsoleSpectacle
Teacher note

The \[ and \] around ANSI color codes in PS1 are a common source of bugs that are hard to diagnose. The symptom: the prompt wraps incorrectly when the command is long — text overwrites the prompt, the cursor jumps to the wrong position. The cause: bash is counting the invisible color code characters as part of the prompt width. The fix: wrap every color code in \[ \]. If a student has this bug, help them fix it directly — it is a good teaching moment about how the terminal calculates screen position.

The screenshot task is genuinely useful at the end of the course. In session 96 (demo day), comparing their terminal at week 11 to their final environment shows visible, concrete growth. Keep a copy yourself — it is good material for a final session retrospective.

Session 44
The shell environment — variables and PATH
5 minquestion
20 minconcept
30 minexercise
5 minclose
Warm-up · 5 min

Ask: when you type ls, bash runs the ls program. But where is ls? How does bash know where to look? It is not built into bash — ls is a separate executable file. Bash searches a list of directories. That list is the PATH variable. Today we understand what environment variables are, how PATH works, and how to add your own scripts to it.

Core concept · 20 min

Environment variables are named values available to bash and to any program bash launches. They carry configuration information: where your home directory is, what your username is, which editor to open for text input, what language to use for messages. Programs read these variables to adapt their behavior without requiring explicit command-line flags every time.

Setting a variable: NAME=value creates a shell variable — visible to bash, not to programs it launches. export NAME=value creates an environment variable — visible to bash and to all programs launched from this bash session. printenv lists all current environment variables. printenv NAME shows the value of one specific variable. echo $NAME also reads a variable's value.

Key built-in environment variables
PATH — colon-separated list of directories to search for commands HOME — your home directory (/home/username) USER — your username SHELL — path to your shell (/bin/bash) EDITOR — preferred text editor (used by git, cron, etc.) LANG — language and locale (en_US.UTF-8) TERM — terminal type (xterm-256color) PS1 — your shell prompt (covered in session 43)

PATH in detail. When you type a command, bash searches each directory in PATH from left to right. The first matching executable wins. If no match is found: "command not found". echo $PATH shows the current PATH — it is colon-separated: /usr/local/bin:/usr/bin:/bin:/usr/sbin:/sbin. which commandname shows which executable bash would run for that command — which directory in PATH it found it in.

Adding to PATH. To add ~/scripts to PATH: export PATH="$HOME/scripts:$PATH". The new directory goes at the front (searched first). $HOME/scripts expands to your actual home path. $PATH appends the existing PATH. To make this permanent: add the export line to ~/.bashrc. Temporary (in terminal only): the change vanishes when the terminal closes. Permanent (in .bashrc): the change applies to every new terminal session.

Exercise · 30 min
  1. Run printenv | sort | less. Browse all environment variables. Find PATH, HOME, SHELL, USER, LANG, TERM. Write what each one contains on your machine.
  2. Run echo $PATH. Count the directories separated by colons. Which one contains most commands? (hint: ls /usr/bin | wc -l)
  3. Run which ls, which bash, which nano, which python3. What does 'which' tell you? What happens if you run which nonexistentcommand?
  4. Create ~/scripts/greet (no .sh extension) containing: #!/bin/bash and echo "Hello from $USER on $(hostname)". Make it executable. Try running it by just typing greet — it will fail with "command not found." Why?
  5. Add ~/scripts to your PATH temporarily: export PATH="$HOME/scripts:$PATH". Try greet again. Does it work now?
  6. Open a new terminal — does greet still work? Why not? Now add the export line to ~/.bashrc. Open another new terminal. Does it work now?
  7. Set export EDITOR=nano in ~/.bashrc. What programs will this affect?
Vocabulary
environment variableexport printenvPATH which$HOME $USER $SHELL EDITORtemporary vs permanent command not found
Teacher note

Tasks 4–6 form a deliberate sequence: the command fails, they fix it temporarily, they open a new terminal and it fails again, they make it permanent, they open another new terminal and it works. This sequence teaches the temporary vs permanent distinction more effectively than any explanation. Students who experience all three steps in sequence rarely confuse them again.

The EDITOR variable (task 7) matters more than it seems. Git uses $EDITOR for commit messages. Cron uses it for editing crontab. sudo -e uses it for safe file editing. Any program that needs to open a text editor uses this variable. Students who set it to nano will never be unexpectedly dropped into vi when they make a git commit — which is otherwise a confusing experience for beginners.

Session 45
Dotfiles — your configuration as code
10 minscenario
20 minconcept
25 minexercise
5 minclose
Warm-up · 10 min

Ask: if this machine broke tonight and you had to set up a new one tomorrow, how long would it take to get back to exactly the way you have it now — your aliases, your prompt, your KDE theme, your shortcuts, your installed packages? Take honest estimates. For most students it would be hours of clicking and trying to remember. Tell them: professionals who manage this well can rebuild their environment in under 10 minutes. This session explains how.

Core concept · 20 min

Dotfiles are the hidden configuration files in your home directory that define your working environment. The name comes from the leading dot that makes them hidden. Every customization you made in week 11 was stored in a dotfile. Every alias you defined is in a dotfile. Every readline binding you set is in a dotfile.

The critical dotfiles for this course:

Important dotfiles — location and purpose
~/.bashrc runs every new interactive terminal session ~/.bash_profile runs once at login (often just sources .bashrc) ~/.profile more portable alternative to .bash_profile ~/.inputrc readline configuration (keyboard bindings) ~/.nanorc nano editor configuration ~/.gitconfig git settings (name, email, aliases) — Phase 5 ~/.config/ directory for modern app configuration ├── konsole/ Konsole profiles ├── kdeglobals KDE-wide settings └── plasma*/ Plasma shell config ~/.local/share/ user-specific application data

The key insight about ~/.bashrc vs ~/.bash_profile: bash reads these at different times. ~/.bash_profile is read when you log in — either at a text console or via SSH. ~/.bashrc is read when you open a new terminal window within an already-logged-in session. On desktop Linux, you almost always want your configuration in ~/.bashrc because you are starting terminals from inside a running desktop session. A common pattern in ~/.bash_profile: [ -f ~/.bashrc ] && source ~/.bashrc — this makes both cases use the same configuration.

The portable environment concept: because your entire working environment is defined by text files in your home directory, you can copy those files to any other Linux machine and have an almost identical environment. This is why developers maintain "dotfiles repositories" — a git repository containing all their configuration files, allowing one-command setup on any new machine. We introduce the concept now and connect it to git in Phase 5.

Exercise · 25 min
  1. Run ls -la ~ and list every dotfile and dotdirectory. Count them. How many exist that you did not create manually?
  2. Open each of these and read them — write one sentence describing what each contains: ~/.bashrc, ~/.bash_history, ~/.profile.
  3. Run ls ~/.config. Open 2 application directories and read whatever configuration files are inside. What format are they? (INI, TOML, JSON, or something else?)
  4. Make your "important dotfiles" list: which 5 files would you most miss if they were deleted? Write them down with reasons.
  5. Create ~/dotfiles/. Copy your ~/.bashrc and any other dotfiles from your list into it. This is your backup. Verify the copies are current with diff: diff ~/.bashrc ~/dotfiles/.bashrc should produce no output.

dotfile~/.bashrc ~/.bash_profile~/.profile ~/.inputrc~/.config diff commanddotfiles backup portable configuration
Teacher note

The diff command introduced in task 5 is a small but important addition. diff file1 file2 shows the differences between two files — no output means they are identical. Students who use diff to verify their backups are accurate are developing a professional verification habit. Diff will return properly in Phase 5 when comparing script versions and config changes.

Students who ask about git for dotfiles are asking exactly the right question. Tell them: git is precisely the right tool for this, and we cover git in Phase 5. For now, a simple directory copy is sufficient and teaches the underlying concept. The git refinement comes later.

Session 46
Editing .bashrc — aliases, functions, and prompt
5 minsurvey
15 minconcept
35 minexercise
5 minclose
Warm-up · 5 min

Quick survey: what command does each student type most often? Collect 10–15 answers on the board. Sort them by frequency. These are the strongest alias candidates. The commands that appear multiple times across the class become the session's alias examples — they are genuinely useful, not invented for the exercise.

Core concept · 15 min

Aliases replace one command string with another. Syntax: alias name='replacement'. The replacement is substituted literally when you type the alias name. Aliases can replace a command with a safer or longer version: alias rm='rm -i' makes rm always ask before deleting. Aliases can abbreviate long commands: alias update='sudo apt update && sudo apt upgrade'. Aliases cannot take arguments — for that, use functions.

Functions take arguments, run multi-step logic, and return values. Syntax:

function name() {
    local variable="$1"   # local keeps variable inside the function
    commands
}

Inside a function, $1, $2 etc. refer to the function's arguments, not the script's. local declares a variable that only exists inside the function — always use local for function variables to avoid side effects. Call a function by name: name argument1 argument2. The most useful pattern: functions that combine multiple steps into one action. The mkcd pattern — create a directory and immediately enter it — is the canonical example:

function mkcd() {
    mkdir -p "$1" && cd "$1"
}

PS1 color codes — a practical reference. Format: \[\033[CODEm\]text\[\033[0m\]. Codes: 0=reset, 1=bold, 31=red, 32=green, 33=yellow, 34=blue, 35=magenta, 36=cyan, 37=white. The \[ and \] brackets tell bash these characters are non-printing — critical for correct cursor positioning on long lines.

Exercise · 35 min

Open ~/.bashrc in nano. Work through each task, saving and sourcing after each group:

  1. Add at least 5 aliases you will actually use. Required: ll='ls -la' and ..='cd ..'. Choose 3 others based on your most-used commands from the warm-up. Test each one.
  2. Add a safer rm: alias rm='rm -i'. Test it — what changes? Note: this only applies to interactive terminal use. Scripts that call rm still get the real rm. Why?
  3. Write the mkcd function. Test it: mkcd ~/test-function/nested/path should create all directories and leave you inside the deepest one. Verify with pwd.
  4. Write a function called extract() that takes a compressed file as $1 and extracts it using the right tool based on the file extension. Use if/elif to check: .tar.gz → tar -xzf, .zip → unzip, .tar.bz2 → tar -xjf. Test with a real archive.
  5. Finalize your PS1 prompt. Requirements: must show current directory, must use at least one color, must not break when the line is long (test with a deeply nested path). Save and open a new terminal to verify.
  6. Open a completely new terminal (not just source). Test every alias and function. Fix any that do not work.
Vocabulary
aliasalias rm='rm -i' functionlocal variable mkcd patternextract function PS1 color codesnew terminal test && (and operator)
Teacher note

Task 2 — alias rm='rm -i' — raises an important question: why do scripts not respect this alias? Because aliases are only expanded in interactive shells, not in scripts. When bash runs a script, it starts in non-interactive mode and does not load aliases. This is the correct behavior — scripts should be explicit and not depend on interactive shell configuration. This distinction between interactive and non-interactive bash will matter in Phase 5 when debugging scripts that work in the terminal but not as cron jobs.

The extract function (task 4) is genuinely useful and connects back to the tar introduction in session 32. Students who write it well will use it for the rest of the course. Students who struggle with the if/elif structure will hit a preview of session 50's conditionals — that is fine. Let them struggle and then help them — struggling here primes them to understand the concept lesson better.

Session 47
Shell configuration depth — inputrc and nanorc
5 minlive demo
20 minconcept
30 minexercise
5 minclose
Overview

Two configuration files that almost no beginner documentation mentions, but that experienced terminal users configure immediately. The improvements are small individually. Combined and used daily, they meaningfully reduce friction.

Warm-up · 5 min

Live demo before explaining anything. Type a partial command from history — say, the first two letters of sudo apt update. Press the up arrow. It goes to the previous command, not the last command starting with "su". Now add this to ~/.inputrc:

"\e[A": history-search-backward
"\e[B": history-search-forward

Open a new terminal. Type "su" and press the up arrow. It now finds the last command starting with "su". Do this live, in front of the class, before explaining what readline is. Let the effect speak first, then explain the mechanism.

Core concept · 20 min

Readline is a library that provides line-editing capability in bash and many other terminal programs. When you use the arrow keys, tab completion, Ctrl+A to go to line start, Ctrl+E to go to line end, Ctrl+U to clear the line — all of that is readline. ~/.inputrc is readline's configuration file. Changes to it affect bash and any other program that uses readline (which includes many: python interactive mode, mysql client, many others).

Key inputrc settings:

Recommended ~/.inputrc settings
"\e[A": history-search-backward # up arrow = search back "\e[B": history-search-forward # down arrow = search forward set completion-ignore-case on # Tab: case insensitive set show-all-if-ambiguous on # Tab once shows completions set colored-stats on # Tab shows file types in color set mark-symlinked-directories on # adds / to symlinked dirs

nano configuration: ~/.nanorc controls nano's behavior. On some systems, the global nanorc at /etc/nanorc provides defaults — your ~/.nanorc overrides or adds to them.

Recommended ~/.nanorc settings
set autoindent # new line matches indentation of previous set linenumbers # show line numbers on the left set mouse # click to position cursor set tabsize 4 # tab key = 4 spaces set softwrap # wrap long lines visually (not in file) set indicator # scrollbar on the right side set zap # delete selected text when you type

Syntax highlighting in nano: nano >= 4.0 supports syntax highlighting for many file types out of the box. Check your version with nano --version. If you have 4.0+, syntax highlighting for .sh, .py, .html and many others should work automatically without configuration.

Exercise · 30 min
  1. Create or open ~/.inputrc. Add all six settings from the reference above. Open a new terminal and test each one:
    • History search: type "ls" and press up — does it find your last ls command?
    • Case-insensitive completion: type "ls /H" and press Tab — does it complete to /home?
    • Colored stats: press Tab twice after a partial path — are directories and files shown in different colors?
  2. Create or open ~/.nanorc. Add the settings from the reference. Open a file in nano — verify: line numbers visible, mouse click moves cursor, tab inserts 4 spaces (test with the Tab key).
  3. Open a .sh file in nano. Does syntax highlighting apply? How can you tell?
  4. Test autoindent: open a new file, write a line with leading spaces, press Enter — does the new line match the indentation?
  5. Write a one-paragraph comparison: how does your terminal experience now compare to how it was at the start of week 11? What is concretely different and faster?
  6. Update your dotfile backup: copy the updated .bashrc, .inputrc, and .nanorc to ~/dotfiles/. Run diff on each to verify they are current.
Vocabulary

The history-search-backward binding reliably gets the strongest reaction of any single configuration change in Phase 3. The reason this is not the default in bash is historical: bash's default behavior is consistent with the readline defaults set in the 1980s, and changing them would break scripts and tutorials that assume the old behavior. This is a good moment to discuss backwards compatibility as a constraint on software design — the cost of making something easy for new users vs the cost of breaking existing users.

Session 49
What a script is and why it exists
10 minmotivation
20 minconcept
25 minexercise
5 minclose
Warm-up · 10 min

Ask: remember session 32 — the messy downloads folder? You ran 20 commands to organize it. If you had to do that every week for a year, what would you do? The natural answer is: find a way to run those commands automatically. That is exactly what a script is — the commands you already know, in a file, run in sequence. Ask students to open their commands.log from session 32. How close is it to a script already? For most of them: one or two changes away.

Core concept · 20 min

A shell script is a text file containing bash commands. When you run it, bash reads the file from top to bottom and executes each line in sequence, exactly as if you had typed them in a terminal. The two differences from the terminal: speed (the machine types faster than you), and repeatability (it runs identically every time).

The shebang line — #!/bin/bash — must be the absolute first line of every script. It tells the OS which interpreter to use to run the file. Without it, the OS guesses, and sometimes guesses wrong (using /bin/sh, which is not always bash and lacks some bash features). Always include it. Always put it first, with no blank lines or spaces before it.

Command substitution — $(command) — runs a command and uses its output as a value. echo "Today is $(date)" runs date, captures its output, and inserts it into the string. echo "There are $(ls | wc -l) files here" counts files and inserts the number. Command substitution is how scripts gather information dynamically — the current date, the number of files, the output of any command — and use it in later steps.

Debugging with bash -x: running bash -x scriptname.sh prints each command before executing it, with a + prefix. You see exactly what bash is doing, with variables expanded. This is the first tool to reach for when a script does not behave as expected.

Three questions that justify writing a script: will I do this more than once? Is it more than 3 steps? Would I trust myself to remember every step correctly under pressure? Yes to any: script it. The script is also documentation — it records exactly what you did, in order, permanently.

Exercise · 25 min
  1. Look at your commands.log from session 32. What would need to change to make it a runnable script? (Add shebang, fix paths, handle variables.) Write the answer before touching the file.
  2. Create ~/scripts/info.sh with nano. Write: 
    #!/bin/bash
echo "User: $USER"
echo "Home: $HOME"
echo "Date: $(date)"
echo "Directory: $(pwd)"
echo "Files here: $(ls | wc -l)"
echo "Uptime: $(uptime -p)"
    Save, chmod +x, run it. Understand every line.
  3. What does $(date) do? What does $(pwd) do? Try: echo "Shell: $(echo $SHELL)" — is this useful or redundant? Why?
  4. Remove the shebang from info.sh and run it. What changes? Put it back.
  5. Run bash -x info.sh. Read the output carefully — what does the + prefix indicate? How does it differ from running the script normally?

Students who realize they were already halfway to scripting in session 32 feel capable rather than intimidated. Session 50
Variables, input, and conditionals
5 minneed
25 minconcept
25 minexercise
5 minclose
Warm-up · 5 min

info.sh from session 49 always does the same thing — it shows your info. What if you wanted it to show info about a different user? What if you wanted it to behave differently depending on whether a file exists? These two needs — flexible input and conditional behavior — are what this session adds. The result: scripts that are tools, not just recordings.

Core concept · 25 min

Variables. NAME=value — no spaces around the =. Read with $NAME or ${NAME}. Always quote variable references in double quotes: "$NAME" not $NAME — this prevents word splitting when the variable contains spaces. Special variables: $1 $2 $3 — positional arguments passed to the script when called. $0 — the script's own name. $# — the number of arguments. $@ — all arguments as separate words. $* — all arguments as one string. $? — the exit code of the last command (0 = success, non-zero = failure).

Reading user input. read -p "Prompt text: " VARIABLE — prints the prompt and waits for the user to type, storing the result in VARIABLE. Always provide a prompt (-p) so the user knows what to enter. Combine with checking for empty input: if the user presses Enter without typing, $VARIABLE will be empty.

Conditional syntax — spaces are mandatory
if [ condition ]; then commands elif [ other_condition ]; then other_commands else fallback_commands fi File tests: -f file (is a regular file) -d dir (is a directory) -e path (exists, any type) -r file (readable) -w file (writable) -x file (executable) String tests: -z "$str" (empty string) -n "$str" (non-empty string) "$a" = "$b" (strings equal) "$a" != "$b" (strings not equal) Number tests: $a -eq $b (equal) $a -ne $b (not equal) $a -gt $b (greater than) $a -lt $b (less than)

The spaces inside [ ] are mandatory — [ is actually a command, not syntax. if ["$NAME"="hello"] is wrong. if [ "$NAME" = "hello" ] is correct. The spaces separate the [ command from its arguments. Missing spaces produce a cryptic syntax error. Always check spacing first when if statements fail.

Exercise · 25 min

Write three scripts in ~/scripts/. Make each executable. Test edge cases — what happens with no arguments, wrong types, missing files:

  1. greet.sh — takes a name as $1. If $1 is empty (test with -z), use read to ask for a name. Then print: "Hello NAME, today is DATE." using command substitution for the date.
  2. check-path.sh — takes a path as $1. Checks: is it a regular file? A directory? Does it not exist? Prints the appropriate message for each case. If no argument given, prints a usage message and exits with code 1.
  3. safe-delete.sh — takes a filename as $1. If the file does not exist, print an error and exit. If it exists, print "Delete FILE? (y/n)" and read the answer. Delete only if the answer is y or Y. Print "Cancelled" otherwise. Test all cases.
Vocabulary
$1 $2 (positional args)$# $@ (all args)$? (exit code) read -pif [ ] then fi -f -d -e (file tests) -z -n (string tests) exit 1 (error exit)usage message
Teacher note

The usage message pattern in check-path.sh — printing how to use the script and exiting with code 1 when no argument is given — is a professional habit worth naming explicitly. Every command-line tool that expects arguments should check for them and print clear usage instructions if they are missing. Students who build this habit from their first scripts write tools that other people (and future-them) can actually use without reading the source code.

Exit codes (the exit 1 in the usage message, and $? in general) deserve a brief extra minute. The convention: 0 means success, anything else means failure. The specific non-zero value can encode what went wrong — 1 is generic failure, 2 is misuse of shell builtins, 127 is command not found. Scripts that set exit codes correctly compose correctly in pipelines and conditionals.

Session 51
Loops and functions
5 minscale problem
20 minconcept
30 minexercise
5 minclose
Warm-up · 5 min

Present this problem: you have 50 image files that need to be renamed from IMG_001.jpg, IMG_002.jpg etc. to photo_001.jpg, photo_002.jpg. Without loops: 50 mv commands, typed individually or with a complex wildcard. With a for loop: 4 lines. That ratio — 50 commands collapsed into 4 lines — is what loops do.

Core concept · 20 min

For loops.

for variable in list; do
    commands using $variable
done

The list can be: a literal list of words (for color in red green blue), a wildcard expansion (for file in *.jpg), or the output of a command (for line in $(cat list.txt)). The loop runs once for each item, with $variable set to that item. seq 1 10 generates numbers 1 through 10 — useful for for i in $(seq 1 10). {01..50} is brace expansion — generates 01, 02, ... 50 with zero-padding.

While loops.

while [ condition ]; do
    commands
done

Runs while the condition is true. Useful for: running until a file appears, reading lines from a file (while read line; do ...; done < file), creating interactive menus that repeat after each action.

Functions.

function name() {
    local var="$1"
    commands
    return 0
}

Call with name arg1 arg2. Inside the function, $1 $2 refer to the function's arguments (not the script's — this surprises people). local declares a variable that only exists inside the function. return N sets the exit code. To return a string value, use echo "value" and capture with result=$(name arg).

The logging pattern — a function that prepends a timestamp to every output line:

function log() {
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] $1"
}

Use in scripts: log "Starting backup", log "Processing file $f". Scripts that log their actions are dramatically easier to debug and audit.

Exercise · 30 min
  1. Add the log() function to ~/scripts/greet.sh from session 50. Replace all echo statements with log calls. Run the script — how does the output change?
  2. Write rename-photos.sh in ~/sandbox/photos/ (create some test files with touch: touch IMG_{001..010}.jpg). The script loops over all .jpg files and renames them to photo_NNN.jpg where NNN is a zero-padded counter. Run it. Verify the results with ls.
  3. Write backup.sh that takes a source directory ($1) and destination ($2). Loops over every file in source, copies each to destination, and logs each copy. If destination does not exist, create it. Handle missing arguments with a usage message.
  4. Write a menu script: shows 4 options (Disk space, Running processes, Uptime, Quit), reads the user's choice with read, runs the appropriate command, then shows the menu again. Use a while loop. Quit option breaks the loop.
Vocabulary
for loopwhile loop seqbrace expansion {01..10} while read linefunction local variablereturn N log() patternmenu pattern
Teacher note

The zero-padded counter in rename-photos.sh requires a technique worth explaining: printf "%03d" $counter formats a number with leading zeros to 3 digits. Students who discover this independently are doing professional-level shell scripting. Those who do not should be shown the pattern and told it is a common requirement in file renaming and log naming.

The menu script (task 4) introduces a pattern — while loop with read and case statement — that appears in real system administration tools. Students who build this correctly have written something that looks like a real interactive program, not just a sequence of commands. That shift in self-perception matters. Name it: "You just wrote an interactive application. It is simple, but it is real."
